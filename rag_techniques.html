✂️
Adaptive Chunking
NLTK sentence tokenizer + Heading detection + Overlap

Thay vì cắt cứng theo số token, hệ thống chia theo cấu trúc ngữ nghĩa: phát hiện heading để tách section, chia từng sentence bằng NLTK, gộp câu đến khi đạt ngưỡng ký tự, giữ lại 12% overlap giữa các chunk liền kề.

Tại sao không dùng fixed-size? Fixed-size có thể cắt ngang câu hoặc đoạn văn, làm mất ngữ nghĩa. Sentence-based đảm bảo mỗi chunk là đơn vị ngữ nghĩa hoàn chỉnh.
Overlap để làm gì? Đảm bảo context ở ranh giới giữa 2 chunk không bị mất. Câu kết của chunk trước được lặp lại ở đầu chunk sau.
# Params trong chunker.py _TARGET_TOKENS = 400 # chars (Vietnamese ~1.5 chars/token) _MAX_TOKENS = 500 # hard limit _OVERLAP_RATIO = 0.12 # 12% overlap
NLTK Sentence splitting Heading detection Sliding window overlap

Phase 2 — Retrieval

🔍
Hybrid Search
Vector Search + BM25 → RRF Fusion

Kết hợp hai phương pháp tìm kiếm bổ sung cho nhau: Vector search (semantic — hiểu ý nghĩa, bắt được paraphrase) và BM25 (keyword — chính xác với tên riêng, thuật ngữ kỹ thuật). Kết quả từ hai nguồn được gộp bằng Reciprocal Rank Fusion (RRF).

Tại sao kết hợp? Vector giỏi paraphrase nhưng yếu với keyword cụ thể. BM25 ngược lại. Hybrid lấy điểm mạnh của cả hai.
RRF là gì? Mỗi chunk nhận điểm = Σ 1/(k + rank). Chỉ dùng thứ hạng, không cần normalize score — cosine và BM25 không cùng thang điểm.
# RRF score formula rrf[chunk_id] += 1.0 / (k + rank + 1) # Chunk xuất hiện ở top của cả 2 danh sách → điểm cao nhất
BM25 (rank_bm25) Vector search RRF fusion Semantic + Keyword
💭
HyDE — Hypothetical Document Embeddings
LLM tạo đoạn văn giả → embed as "passage" → vector search

Thay vì embed câu hỏi ngắn (thường chỉ 5–10 từ), LLM được yêu cầu viết một đoạn văn giả (3–5 câu) như thể đang trả lời câu hỏi đó. Đoạn văn này được embed với input_type="passage", giúp vector nằm gần với các chunks trong tài liệu hơn.

Vấn đề HyDE giải quyết "FastAPI là gì?" (query vector) ≠ "FastAPI is a modern Python framework..." (passage vector). Khoảng cách embedding lớn → bỏ lỡ kết quả liên quan.
Cách triển khai BM25 vẫn dùng query gốc. Chỉ vector search mới dùng HyDE doc. Nếu LLM lỗi, fallback về query gốc.
# HyDE prompt "Write a concise passage (3-5 sentences) that directly answers the question below. Write as if you are the source document itself." # Sau đó embed với input_type="passage" vec_results = vector_search(hyde_doc, input_type="passage") bm25_results = bm25_search(original_query) # query gốc
HyDE Query expansion Asymmetric embedding Env: HYDE_ENABLED
🧩
Query Decomposition
LLM chia câu hỏi phức tạp → N sub-queries → tìm song song → RRF merge

Với câu hỏi nhiều vế (ví dụ: "So sánh FastAPI và Flask về performance và ease of use"), LLM chia thành N sub-queries độc lập. Mỗi sub-query được search riêng, kết quả được merge bằng Multi-query RRF. Chunk xuất hiện trong nhiều sub-query được ưu tiên.

Tại sao cần? Một câu hỏi phức tạp thường không match tốt với bất kỳ chunk đơn lẻ nào. Tách nhỏ → tìm được nhiều context hơn.
Multi-query RRF Merge N result lists với RRF. Chunk xuất hiện ở top nhiều list → điểm tích lũy cao → được ưu tiên đưa vào reranker.
# Decompose prompt → ["sub-q 1", "sub-q 2", "sub-q 3"] for q in sub_queries: hyde_text = generate_hypothetical_doc(q) # nếu HyDE bật results = hybrid_search(q, hyde_text=hyde_text) all_results.append(results) candidates = multi_rrf(all_results) # merge
Query decomposition Multi-query retrieval Multi-query RRF Env: QUERY_DECOMPOSITION_ENABLED

Phase 3 — Reranking

⚖️
Cross-Encoder Reranking
BAAI/bge-reranker-base (sentence-transformers)

Sau khi hybrid search trả về ~20 candidates, cross-encoder đánh giá từng cặp (query, chunk) cùng lúc — thay vì so sánh vector riêng lẻ như bi-encoder. Chính xác hơn nhiều nhưng chỉ chạy được trên pool nhỏ do complexity O(n).

Phương phápCách hoạt độngĐộ chính xácTốc độ

Cross-encoder (reranker) Xử lý cặp (query, chunk) cùng lúc Cao Chậm hơn O(n)

Phase 4 — Generation

🤖
LLM Generation
NVIDIA NIM — openai/gpt-oss-120b

Top-K chunks sau reranking được ghép thành context block, kèm system prompt yêu cầu LLM chỉ trả lời dựa trên context và cite số trang. Model gọi qua NVIDIA NIM API — OpenAI-compatible, không cần thay đổi client code.

RAG vs Pure LLM Pure LLM có thể hallucinate vì không biết nội dung tài liệu. RAG grounding câu trả lời vào chunks thực tế, giảm hallucination.
System prompt strategy "Always base answers strictly on the context. If context is insufficient, say so clearly. Cite page numbers."

① Parametric Memory
Kiến thức được mã hóa trong trọng số model (model weights)

Là kiến thức LLM học được trong quá trình training — không thể đọc, ghi, hay cập nhật trong runtime. Bao gồm: cách hiểu ngôn ngữ, reasoning, kiến thức thế giới chung. Cross-encoder reranker (bge-reranker-base) cũng thuộc loại này.

Trong project này gpt-oss-120b (answer generation), bge-reranker-base (cross-encoder scoring), nv-embedqa-e5-v5 (embedding)
Hạn chế Không biết nội dung tài liệu cụ thể của user. Có thể hallucinate khi được hỏi về nội dung không có trong training data. RAG giải quyết điều này.
Model weights Static / Read-only gpt-oss-120b bge-reranker-base
🗄️
② Long-term External Memory
Qdrant Vector Store — kiến thức tài liệu, persistent trên disk

Là "bộ nhớ dài hạn" của hệ thống — nơi lưu toàn bộ kiến thức từ tài liệu PDF sau khi ingest. Mỗi chunk được embed thành vector 1024 chiều và lưu kèm payload (text, page, doc_id). Tìm kiếm qua HNSW index với cosine similarity, filter theo document_id.

Ghi (Write) Khi ingest PDF: chunk → embed → upsert vào Qdrant. Persistent trên disk tại ./storage/qdrant/.
Đọc (Read) Mỗi query: embed query → ANN search → trả về top-K chunks có score cao nhất.
# Payload mỗi vector point trong Qdrant { "document_id": "4f248366-...", "content": "FastAPI is a modern...", "page": 3, "type": "text", # hoặc "image" "chunk_id": "..." }
Qdrant HNSW index Persistent on disk Read + Write Semantic memory
⚡
③ Working Memory
BM25 Index — ephemeral, rebuilt per-request từ chunks đã load

BM25 index được xây dựng mỗi lần query từ danh sách chunks tải từ disk — không persist, không cache giữa requests. Là "bộ nhớ tạm" giữ term-frequency statistics trong lúc search, sau đó giải phóng. Nhanh vì corpus nhỏ (vài trăm chunks mỗi tài liệu).

Tại sao không cache BM25? BM25 index gắn với corpus. Nếu tài liệu thay đổi (re-ingest), index cũ sẽ stale. Rebuild mỗi request đảm bảo luôn đúng.
Trade-off Tốn ~10–50ms mỗi request để build index. Chấp nhận được ở document scale nhỏ (~vài trăm chunks). Ở scale lớn cần cache layer.
# Mỗi request: load → build → search → discard chunks = load_chunks(document_id) # từ disk bm25 = BM25Okapi([c["text"].split() for c in chunks]) scores = bm25.get_scores(query.split()) # working mem # sau khi return → GC giải phóng bm25 object
BM25Okapi Ephemeral / Per-request Keyword statistics No persistence
💬
④ Short-term Memory — LLM Context Window
Assembled prompt: System + Retrieved chunks + Chat history + Query

Đây là "bộ nhớ tức thời" được lắp ráp mỗi request và gửi vào context window của LLM. Bao gồm 4 thành phần theo thứ tự: system prompt, retrieved chunks (từ long-term memory), lịch sử chat, và câu hỏi hiện tại của user.

System Prompt
"Answer only from the context. Cite page numbers. If insufficient, say so."
Retrieved Context
Top-5 chunks từ Qdrant + reranker. Nguồn: long-term external memory.
Chat History
N turns gần nhất. Giữ trong React state (frontend). Không lưu DB — mất khi reload.
User Query
Câu hỏi hiện tại.
Giới hạn context window LLM có context limit (vd 128K tokens). Cần chọn K chunks phù hợp — quá nhiều tốn token, quá ít thiếu context. Reranker giúp chọn đúng.
Chat history hiện tại Project lưu history trong React state — ephemeral (mất khi reload). Không có persistent chat DB. Đây là điểm có thể cải thiện.
Context window Ephemeral per-request Chat history (React state) Prompt assembly
